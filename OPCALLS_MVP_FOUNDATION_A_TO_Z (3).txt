OPCALLS / VOICEAGENT PLATFORM
FOUNDATION DOCUMENT FOR MVP
A-to-Z product, architecture, security, billing, provisioning, scalability, and flexibility blueprint
Version 1.0

PURPOSE
This document is the execution foundation for the MVP of a multi-tenant SaaS that lets businesses launch AI voice agents with almost no technical knowledge. The system must feel frictionless to end users, but internally it must be built like a serious control plane: billing-safe, provider-agnostic, auditable, scalable, and easy to evolve.

CORE PRODUCT TRUTH
This is not just an “AI call agent builder.”
This is a zero-config vertical front-desk operating system.

The client experience must feel like this:
choose industry -> enter business basics -> choose number -> pay -> agent goes live -> see ROI and call activity.

The system must be designed so that:
- end users do not touch raw Retell/Twilio configuration
- admin can review and approve before a real deployment goes live
- the platform can later swap Retell for Bland, Vapi, or another AI provider with minimal disruption
- billing and service access are controlled by the platform, not by manual operations
- all risky or high-cost actions are behind entitlement, fraud, and spend controls

====================================================================
C. PRODUCT PILLARS
====================================================================

1. Frictionless onboarding
No technical dashboard during setup. The product should ask business questions, not AI configuration questions.

2. Controlled automation
Automation should be strong, but never blind. Admin review must be possible before production launch.

3. Provider flexibility
All Twilio / Retell / Stripe logic must sit behind adapters so providers can be swapped later.

4. Billing-first safety
Before any costly action, the system must check subscription state, spend limits, fraud score, and entitlements.

5. Vertical depth
A clinic should not get the same generic agent as a salon. Every vertical pack must have distinct flows, language, constraints, and KPIs.

6. Measurable value
The system must show outcome metrics such as bookings, lead captures, estimated revenue, missed-call recovery, and time saved.

====================================================================
G. PROVIDER-AGNOSTIC DESIGN REQUIREMENT
====================================================================

This is one of the most important foundation rules.

Do not let the app talk directly to provider SDKs everywhere.
All provider-specific logic must be wrapped behind internal adapters.

Required provider interfaces:
- BillingProvider
- TelephonyProvider
- AIProvider
- QueueProvider
- StorageProvider
- NotificationProvider

Minimum behavior of these providers:

1. BillingProvider
- create customer
- create checkout session
- create portal session
- parse webhook
- read subscription status
- handle upgrades / downgrades
- support metering later

2. TelephonyProvider
- create tenant account / subaccount
- search numbers
- buy number
- release number
- create trunk
- attach number to trunk
- suspend account
- resume account
- validate webhook
- fetch call metadata

3. AIProvider
- create agent
- update agent
- version prompt config
- import phone number
- bind phone number
- unbind phone number
- create knowledge source
- trigger test call if supported
- validate webhook
- fetch call result metadata

4. QueueProvider
- enqueue job
- schedule retry
- dead-letter failed jobs
- replay job

5. NotificationProvider
- email
- SMS
- internal alerts

Result:
Switching from Retell to Bland or Vapi should mostly mean creating a new provider implementation, not rewriting the entire app.

====================================================================
K. BILLING MODEL AND PAYMENT TRUTH
====================================================================

This must be implemented clearly.

1. Payment truth
- customer pays you through Stripe
- Twilio bills your master account
- Retell bills your provider account
- your platform manages access
- this is not a direct automated split between Stripe and vendors in the base model

2. Billing states
- draft
- trialing
- active
- past_due
- restricted
- suspended
- canceled

3. Dunning flow
A. payment fails
- set past_due
- notify tenant
- start grace timer

B. grace expires
- set restricted
- block outbound
- block advanced settings
- keep billing portal accessible

C. still unpaid
- set suspended
- suspend telephony account / routing
- unbind or redirect AI number behavior

D. payment recovered
- set active
- restore entitlements
- resume providers
- run health check

4. Spend guardrails
Per tenant define:
- soft spend threshold
- hard spend threshold
- max daily minutes
- max concurrent calls
- max outbound attempts per hour
- country allowlist
- call hour window
- fraud threshold

5. Billing UX
Tenant dashboard must show:
- current plan
- included limits
- current usage
- invoices
- payment method
- past_due banner if relevant
- upgrade path

====================================================================
M. PROVISIONING STATE MACHINE
====================================================================

Never implement onboarding as one huge synchronous function.
Use a durable state machine.

Initial launch flow:
- draft
- stripe_customer_created
- subscription_created
- provisioning_queued
- twilio_subaccount_created
- regulatory_check_passed
- number_selected
- number_purchased
- sip_trunk_created
- number_attached_to_trunk
- ai_agent_created
- knowledge_synced
- phone_imported_into_ai
- number_bound
- test_call_passed
- live

Failure branches:
- waiting_retry
- manual_review_required
- rollback_required
- restricted
- failed

Rollback examples:
- number purchased but AI binding failed -> keep number purchased_unbound, retry binding
- AI agent created but KB sync failed -> mark KB pending, deploy minimal prompt if safe
- payment invalid during provisioning -> stop activation and move to restricted

Suspend flow:
- active -> past_due -> restricted -> suspended

Resume flow:
- payment_recovered -> resume_service -> health_check -> active

====================================================================

- store transcript metadata
- classify outcome
- detect success / failure patterns

This creates a flywheel:

====================================================================
Y. SCALABILITY, CONCURRENCY, AND COST CONTROL
====================================================================

Reality:
No platform has infinite concurrency.
The goal is that the architecture degrades safely and scales predictably.

Rules:
1. keep inbound path fast and lightweight
2. move heavy work to queues
3. queue all outbound work
4. rate-limit by tenant
5. set max concurrent actions per tenant
6. separate hot call logging from heavy analytics
7. use reconciliation jobs to detect drift
8. add spend triggers and anomaly detection

Concurrency design:
- inbound call events: lightweight ingestion + async processing
- outbound campaigns: queued and throttled
- provisioning: fully async
- analytics: batched rollups

Future scale options:
- shard heavy tenants to dedicated workers
- isolate high-volume enterprise tenants on separate vendor accounts if needed
